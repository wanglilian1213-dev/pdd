# 正文第一次生成写作规则设计

## 目标

把用户新增的“正文写作规则”接到第一次正文生成这一步里，让 AI 在初稿阶段就按更强的论证方式、表达限制和参考文献要求来写。

这次只改第一次正文生成，不改后面的：

- 字数矫正
- 引用格式修正

## 用户已确认的边界

- 只在“第一次正文生成”时把这些规则一起发给 AI
- 不在程序里额外做硬性句式审查
- 不把同一套规则同步到后面的字数矫正和引用修正

## 当前问题

现在第一次正文生成时，系统发给 AI 的要求很短，主要只有：

- 按大纲写完整文章
- 接近目标字数
- 使用指定引用格式
- 只输出文章内容
- 包含参考文献

用户这次新增的强约束规则，目前没有一起发过去，所以 AI 并不知道这些写作偏好：

- 整篇一次写完
- 所有章节都写出来
- 用段落写，不要项目符号
- 要有明确论点和批判性讨论
- 要给出具体证据
- 用第三人称，但要有清晰立场
- 不要某些特定符号和句式
- 每条参考文献要带链接

## 采用方案

采用“直接加进第一次正文生成提示里”的方案。

也就是：

- 保留当前这一步已有的基础要求
- 在同一个 system prompt 里直接补上用户新增规则
- 不新增新的流程分支
- 不加程序级写作审查器

## 为什么这样做

- 改动最小，风险最低
- 最符合用户原意：这些规则就是“每次写作的时候一起告诉 API”
- 不会误伤后面的字数调整和引用修正
- 如果以后用户继续加正文风格要求，也能继续集中维护在这一步

## 具体改法

### 1. 只改第一次正文生成 prompt

目标文件：

- `server/src/services/writingService.ts`

在 `generateDraft()` 里，把当前 system prompt 扩成更完整的一段，明确加入这些要求：

- 一次性写完整篇文章
- 必须覆盖大纲里的全部章节
- 用段落，不要 bullet point
- 这是批判性、论证性讨论，不是浅层描述
- 论点要和 thesis statement 对齐
- 要给出具体、细致的证据
- 采取明确立场，但保持第三人称
- 兼顾多种观点，再给出更有力的批判性结论
- 避免用户点名禁止的符号和句式
- 每条参考文献都提供正确链接

### 2. 不改后两步

以下两步这次不动：

- `calibrateWordCount()`
- `verifyCitations()`

这样可以保证范围收紧，不把“风格要求”扩散到后续专门做字数和引用修正的步骤。

### 3. 先补测试，再改代码

测试只盯一件事：

- 第一次正文生成时，发给 OpenAI 的 prompt 已经包含这些新增规则

不额外测试：

- AI 最终是否真的每次都完美听话
- 程序是否做了句式审查

因为这两件事不在这次范围内。

## 文档同步

这次改完后，需要同步这些文件：

- `agent.md`
- `PLAN.md`
- `DESIGN.md`
- 新增的这份设计文档
- 新增的实施计划

重点是把“第一次正文生成已加入强写作规则”这件事记下来，避免后面文档和真实行为不一致。

## 验证方式

### 本地

- 先写一个失败测试，确认当前 prompt 里没有这些新规则
- 改代码后重跑测试，确认新规则已进入第一次正文生成 prompt
- 再跑后端 `test`、`lint`、`build`

### 线上

- 推送到 GitHub main
- 确认 Railway 的 `app`、`cleanup`、`拼代代前端` 都是最新
- 由于这次只改正文第一次生成提示，真实线上回归重点看：
  - 任务能正常从大纲确认进入正文生成
  - 主链路没有因为 prompt 修改而报错
