# 拼代代 — 设计蓝图

## 产品是干什么的

拼代代是一个帮用户写英文文章的网页工具。用户上传自己的材料，系统帮忙生成英文大纲和正文，最后还能做"降低 AI 痕迹"的处理。

## 当前线上架构：三层分工

整个系统分成三层，每层只管自己的事：

```
┌─────────────────────────────────────────┐
│  第一层：用户看到的网页（前端）              │
│  技术：React + TypeScript + Tailwind       │
│  部署：Railway 前端服务                       │
│  职责：展示界面、收集用户输入、显示结果       │
│  红线：不做业务判断，不存业务真相             │
└────────────────┬────────────────────────┘
                 │ 调用接口
┌────────────────▼────────────────────────┐
│  第二层：业务处理服务（后端）                │
│  技术：Express (Node.js)                    │
│  部署：Railway app + cleanup                 │
│  职责：处理业务逻辑、调用 AI、管理文件       │
│  红线：不自建登录系统，不让前端自己算账       │
└────────────────┬────────────────────────┘
                 │ 读写数据
┌────────────────▼────────────────────────┐
│  第三层：数据和认证（基础设施）              │
│  技术：Supabase（数据库 + 认证 + 文件存储）  │
│  职责：存数据、管用户账号、存文件、做行级权限门禁 │
│  红线：数据真相只有这一份，不允许其他地方存副本│
└─────────────────────────────────────────┘
```

## 用户操作的完整流程图

```
用户注册
  ↓
Supabase 创建账号
  ↓
Railway 初始化（创建用户档案 + 钱包）← 必须成功才能进工作台
  ↓
用户登录
  ↓
兑换激活码获得积分
  ↓
创建任务 + 上传材料
  ↓
系统生成首版大纲（不收费）
  ↓
用户可修改大纲（最多改 4 次，不收费）
  ↓
用户确认大纲 → 冻结积分 → 开始收费链路
  ↓
正文生成 → 字数矫正 → 引用核验 → 交付整理（封面页 + 正文页 + Reference 新页的 Word、专业 PDF 报告）
  ↓
成功：扣除积分，用户下载文件
失败：退回冻结积分，记录失败原因
  ↓
（可选）用户发起降 AI → 再次冻结积分 → 处理 → 成功扣费/失败退回

（独立功能）文章修改：用户上传已有文章 → 前端单文件增量预估字数（POST /api/revision/estimate）→ 实时显示「预估冻结 X 积分」→ 输入修改要求 → 后端 PDF/DOCX 真解析 + 余额前置校验 → 冻结积分 → Claude Opus 4.6 修改 → 成功按修改后字数结算差额退款 / 失败全额退回

（独立功能）文章评审：用户上传文章 + 可选 rubric/brief → 冻结积分 → GPT-5.4 按学术导师标准模拟评审 → 生成 PDF 报告（总分 + 分维度反馈 + 改进建议）→ 成功扣费 + 差额退款 / 失败全额退回
```

## 核心设计决策（为什么这么做）

### 决策 1：前端不存任何真实业务状态

**为什么**：如果前端自己记"任务到第几步了"，一旦用户刷新页面或换个浏览器，数据就丢了。所以所有状态都存在后端数据库里，前端每次都去问后端"现在到哪了"。

### 决策 2：登录只用 Supabase Auth，不自己做

**为什么**：自己做登录系统太容易出安全问题（密码泄露、被攻击等）。Supabase 是专业做这个的，比我们自己做靠谱得多。Railway 的业务接口也用 Supabase 的登录凭证来验证身份，不搞两套。

### 决策 3：同一时间一个用户只能做一个任务

**为什么**：多任务并行会让状态管理变得极其复杂。第一阶段先保证一个任务跑通跑稳，以后再扩展。而且这个限制在前端、后端、数据库三层都要拦截，防止任何一层漏掉。

### 决策 4：工作台是一个页面多个状态，不是多个页面

**为什么**：用户从头到尾在同一个页面（`/dashboard/workspace`）操作，页面根据任务进度自动显示不同内容。这样用户不会迷路，开发也不会做一堆页面跳来跳去搞混。

补充：公共页面也要有完整兜底。现在首页、登录、注册、充值和 404 页面都属于正式入口，不能再把占位文案、死链接和缺页状态直接暴露给用户。

### 决策 4.5：正式文章题目不让用户额外填，而是在大纲阶段自动生成并锁定

**为什么**：现在用户主要上传的是任务说明、评分标准、参考材料。如果强行再加一个“文章标题”输入框，只会让流程更重，还容易和材料冲突。更稳的做法是：

- 先让系统在大纲阶段基于材料、特殊要求、任务标签自动生成一个正式题目和研究问题
- 后面整条链路都统一用这个正式题目

这样可以避免“第一个上传文件名直接变成最终文章标题”这种错误一路传到最后。

### 决策 5：收费阶段失败必须自动退款

**为什么**：用户花了钱但没拿到结果，必须立刻把钱退回去。不能让运营手动退，因为运营可能忙、可能忘。系统必须自动处理，而且退款必须有流水记录，账目要对得上。

### 决策 6：第一阶段不做完整运营前台页面，但要预留运营接口

**为什么**：完整后台页面很费时间。第一阶段先把运营需要的后端接口和数据库操作留好，必要时可以直接用 Supabase 和 Railway 配合处理，后面再决定要不要补页面。

### 决策 7：涉及钱和任务状态的关键动作，尽量收进数据库里一次做完

**为什么**：像“兑换激活码”“确认大纲开始收费”“发起降 AI”这种动作，一旦被连续点两次，或者两个请求同时撞上来，就很容易出现重复扣费、状态错乱、半成功半失败。现在这类关键动作统一收进数据库函数里，一次完成检查条件、改余额、写流水、改任务状态，避免中途卡住。

### 决策 8：配置、审计、监控都要有“不会拖垮主流程”的兜底

**为什么**：配置缓存、审计日志、错误监控这些东西很重要，但不能反过来把主功能拖死。现在的原则是：

- 配置可以短时缓存，减少每次都打数据库
- 审计日志尽量记录，但写失败不能拦住用户动作
- 错误监控是可选开关，不配 `SENTRY_DSN` 时系统照常运行

## 数据库核心表（简化说明）

| 表名 | 存什么 |
|------|--------|
| `user_profiles` | 用户基本信息（昵称、状态等） |
| `wallets` | 用户钱包（余额） |
| `credit_ledger` | 积分流水（充值、消费、退款的每一笔记录） |
| `recharge_codes` | 激活码（面值、状态、谁用了） |
| `tasks` | 任务主记录（当前阶段、状态、失败原因等） |
| `task_files` | 任务相关的文件（上传的材料、生成的结果） |
| `revisions` | 文章修改记录（修改要求、状态、结果文本、冻结积分） |
| `revision_files` | 修改相关的文件（上传的材料和修改后的输出文件） |
| `scorings` | 文章评审记录（状态、场景、总分、结果 JSON、冻结与结算积分、失败原因） |
| `scoring_files` | 评审相关的文件（上传的材料 + 生成的 PDF 报告），含 GPT 识别的角色和精确字数 |

补充规则：

- 面向用户的业务表现在都开了 RLS（行级门禁），用户只能看到自己的数据
- `recharge_codes`、`system_config` 这类敏感表不允许公开身份直接读取，只能走后端 service role
- 新增 `audit_logs` 表，用来留存关键动作的结构化审计记录
- `tasks.course_code` 用来保存自动提取出来的课号；提不出来时允许为空，不阻断后续交付
- `tasks.paper_title` 和 `tasks.research_question` 用来保存大纲阶段自动生成并锁定的正式题目和研究问题；后面正文、封面、下载文件名、报告标题都优先用它们
- `scorings` 表有部分唯一索引 `WHERE status='processing'`，保证同一用户同时只能有一个评审在跑
- `scorings.input_word_count` 存所有上传文件精确字数总和（冻结依据）；`scorings.scoring_word_count` 存 GPT 识别为 `article` 文件的精确字数（结算依据，`clamp(0, input_word_count)` 兜底）
- `scoring_files.hinted_role` 是后端按文件名预判的角色（`article/rubric/brief/unknown`）；`scoring_files.detected_role` 是 GPT 按内容最终裁决的角色（`article/rubric/brief/other`）；两者可能不同，以 GPT 裁决为准

## 文件存储规则

所有文件存在 Supabase Storage 里，按任务 ID 分文件夹。用户上传的材料和系统生成的结果文件都在这里，通过签名链接给用户下载，链接有过期时间。

当前交付文件规则：

- 正文：正式 `Word (.docx)`，固定论文样式输出；第 1 页是封面（课号 + 任务标题），正文从第 2 页开始，`Reference` 必须另起一页
- 引用核验报告：正式 `PDF (.pdf)`，固定报告样式输出，并按真实内容自动换页，不能再出现文字重叠
- 降 AI 后文章：继续是 `Word (.docx)`，并复用同一套正式论文排版
- 任务列表里的单个“下载”按钮固定代表“主文稿下载”，优先顺序是：降 AI 后文章 → 最终正文 → 引用核验报告
- 文章评审 PDF 报告：存在 Storage 的 `scorings/<scoring_id>/report-*.pdf` 路径下，`scoring_files.expires_at` 固定为创建时间 + 3 天；前端通过签名链接按需下载，`scorings` 表记录永久保留供用户查看历史

补充标题和正文清洗规则：

- 正式题目和研究问题必须在第一次大纲生成时一起产出并落库
- 上传文件名只允许继续当“任务标签/默认名称”，不能再直接决定最终交付题目
- 任务要求材料决定正文主题；评分标准、rubric、写作指南只负责补充结构、写作、引用和评分要求，不能替代任务本身的主题
- 如果任务要求材料没有给唯一题目，只给了范围、方向或几个可选项，允许模型自己定一个具体题目，但这个题目必须严格落在任务要求允许的范围里，不能跑去写“怎么写报告”这种元话题
- 这套主题判断不是程序硬编码“哪份文件一定是什么”，而是把规则讲给模型：任务要求材料决定主题；评分标准和写作指南只能辅助；如果材料只给范围而没有唯一题目，允许模型在任务允许范围里自己定具体题目
- 任务要求规则先于大纲和正文：系统要先从任务要求文件里提取字数和引用格式；如果文件里没有，就默认 `1000` 字和 `APA 7`
- 系统再按统一规则换算出：
  - 最少引用数量：每 `1000` 字 `5` 条，向上取整
  - 章节数量：`1000` 字 `3` 章，每多 `1000` 字加 `1` 章，向上取整
- 这两个换算结果必须和字数、引用格式一起落库，后面整条链路都只认这一份统一任务要求结果
- 大纲可用性检查在判断章节数量时，必须按真实多行章节标题计数，不能先把换行压成一行再判断
- 确认大纲只负责“确认开始写正文”，不再允许临时覆盖字数和引用格式
- 交付时统一优先用正式题目；只有老任务还没补出正式题目时，才回退到旧任务标题，并先去掉 `.txt/.pdf/.docx/.doc` 这类真实文件后缀
- 封面标题和正文第一页第一行如果只是直引号 / 弯引号或普通空格差异，也必须当成同一个标题处理，不能重复印两次
- 写正文的 3 个整篇重写步骤（第一次正文生成、字数矫正、引用修正）都要明确禁止 Markdown 样子输出
- 字数矫正这一步只按“正文主体”计数，不把封面标题、正文标题、`References` 标题和参考文献条目算进目标字数；文内引用仍然算正文
- 字数矫正的硬范围固定为目标字数正负 `10%`，并且要把“very strict rule, must follow”明确告诉接口
- 字数矫正只在自己这一段内部消化问题：如果第一次没压进范围，就继续重试，最多 `5` 次；第 `5` 次仍未命中范围时，当前版本继续往后走，这次不顺手改后续交付判断
- 最终导出 Word 前，仍然要再做一次格式兜底：把残留的 `*斜体*`、`**加粗**`、`# 标题` 这类 md 痕迹尽量转成真正的 Word 样式，而不是简单把符号删掉
- 如果大纲里还是 `[Research Question]` 这种占位词，或题目明显还是评分标准名 / 说明文件名，系统会先自动修一次；修不回来才在大纲阶段失败，不会直接放行到正文
- 大纲生成后还要再做一次“主题有没有被评分标准 / 写作指南带偏”的复查；如果跑偏，先自动修一次，修不回来才在大纲阶段失败
- 如果第一次正文是在“请先给题目/研究问题”这种拒答，或者完全没有文内引用、没有参考文献，系统不会立刻终止，而是会把材料 + 正式题目 + 研究问题 + 当前大纲一起回塞给模型自动重跑一次
- 正文链路不再只查“有没有引用”，还要继续围绕统一任务要求检查：
  - 引用数量够不够
  - 是否全部来自 `2020` 年之后的学术论文
  - 是否混入 book
  - 引用格式是否和任务要求一致
- 字数矫正和引用修正只能在“当前文章已经是正常版本”的前提下继续；如果它们把文章改坏，就退回最后一个正常版本继续交付
- 正文超时规则统一调整为：正文初稿单次最多 `30` 分钟；字数矫正每次最多 `15` 分钟；引用修正每次最多 `20` 分钟；降 AI 继续按约 `10` 分钟处理；卡住任务的总兜底默认改成 `45` 分钟
- 历史已完成任务如果只是文件名或排版脏，统一通过“只重做交付文件”的补救流程修复；如果内容本身就是坏的，或者正文主题已经被评分标准 / 写作指南带偏，要走“修大纲 / 重跑正文但不重复收费”的补救流程

## 工作台状态规则

- `outline_ready` 表示“系统已经把大纲准备好，正在等用户确认”，这是正常等待用户操作，不属于后台卡死
- 第 6 步只是“交付阶段”，里面要区分两种状态：
  - `delivering + processing`：系统还在整理最终文件，页面继续显示处理中
  - `completed + completed`：交付真的完成，页面才显示下载卡片和降 AI 入口
- 第 7 步只代表“降 AI 已经真的启动或已经完成”，不能拿它当“正在尝试启动”的过渡页

## 文章评审（Scoring）业务规则

文章评审是 2026-04-15 上线的独立功能，路由 `/dashboard/scoring`，和主写作 / 文章修改互不阻塞。

- 计费：`0.1 积分/word`（由 `system_config.scoring_price_per_word` 控制，支持小数，在 `opsService.validateConfigValue` 有对应小数校验分支）
- "word" 定义：汉字、假名、韩文等 CJK 字符每个算 `1`；西文去掉 CJK 字符后按空格切词；CJK + 西文相加，不引入中文分词依赖
- 冻结：按所有上传文件的精确字数总和冻结；结算：按 GPT 识别为 `article` 角色的文件的精确字数累加，`clamp(0, input_word_count)` 兜底，差额自动退款
- 文件名对齐：GPT 回写的 `detected_files.filename` 走 `normalize()`（trim + toLowerCase + 去路径前缀）再和上传文件匹配；命中 0 条就 fallback 按 `input_word_count` 全额结算（保守策略）
- 前置拒绝：扫描件 PDF（pdf-parse 抽文字为空或 ≥90% 私用区字符）、所有文件都是图片、不支持扩展名（`.doc/.rtf/.odt` 等）都在上传阶段拒绝，不冻结积分
- 并发控制：`scorings` 表有部分唯一索引 `WHERE status='processing'`，同一用户同一时间只能有一个评审在跑
- AI 调用：OpenAI Responses API，`gpt-5.4`，reasoning effort `high`，**不加入 `stagesWithWebSearch`**（评审不联网搜索，避免文章数据外流）；单次超时 20 分钟
- JSON 校验：GPT 返回结果先走 `parseScoringJson`（纯 JSON → 退化为正则抽 `{...}`），再走 `validateScoringJson`（overall_score 0-100、dimensions 1-10 个、weight 和在 [95,105] 容差、top_suggestions 3-6 条、detected_files 有 filename+role）；校验失败最多重试 1 次，两次都错则全额退款 + 标 `failed`
- 分数锚点：百分制 0-100，SYSTEM prompt 里明确锁死 75-84 为"良好（符合要求即应落此区间）"；任何达到作业基本要求的文章不允许 <75；上线前必须用 1000-1500 字范文验证过分数不会被压到 75 以下才能发布
- 容忍度：少于 3 个拼写错误、语法正确的长句、非顶刊但合法期刊引用、语态风格偏好都写进 prompt 的容忍度清单，GPT 不得据此扣分
- 结算顺序：和文章修改同样严格 —— `settleCredits` 必须在 PDF 生成、storage 上传、`scoring_files` 写入、`scorings` 状态更新全部成功之后才能调；任何结算之前的失败先 `refundCredits` 再标 `failed`
- 卡死回收：`scorings.processing` 超过 `stuck_task_timeout_minutes`（默认 45 分钟）由 `cleanupRuntime.cleanupStuckScorings` 自动 refund + 标 failed
- 前端轮询：首次 `5s`，每次 `+3s` 到 `15s` 封顶，总超时 `25` 分钟
- 交付：PDF 报告（pdfkit，Times-Roman 12pt，封面 + 总评 + 改进建议清单 + 分维度卡片 + 检测文件清单，按真实内容自动分页），存储 3 天过期；`scorings` 表永久保留供用户查看历史
- 历史入口：`/dashboard/tasks?tab=scorings` 是第 3 个 tab，懒加载（切到才请求、同 tab 切回用缓存）

## 环境变量

敏感信息（数据库地址、AI 密钥、Supabase 密钥等）全部通过环境变量注入，不允许提交到公开仓库。参考各目录下的 `.env.example` 文件。

补充说明：

- 主写作链路当前统一通过 `OPENAI_MODEL=gpt-5.4` 读取模型名
- 文章修改功能和图表增强走 Anthropic Claude API（`claude-opus-4-6`，开启 extended thinking），图表后压缩走 `claude-sonnet-4-6`；客服聊天、降 AI 辅助处理（压缩/格式修复/润色）已改走 OpenAI GPT-5.4（reasoning effort `high`）；通过 `ANTHROPIC_API_KEY` 和可选 `ANTHROPIC_BASE_URL` 配置
- 降 AI 当前单独走 Undetectable Humanization API，不再复用 OpenAI
- 降 AI 固定参数：`model=v11sr`、`strength=More Human`、`readability=University`、`purpose=Essay`
- 第一次正文生成会额外带上更强的写作规则，明确要求整篇一次写完、所有章节都写、只用段落、不用项目符号、强调批判性论证和具体证据；后续字数矫正和引用修正暂时不复用这套规则
- 正文最终不再直接把 AI 输出原样塞进文件，而是由后端统一排成正式 Word 样式
- Word 交付由程序固定排版，不让 AI 直接吐 `.docx`，这样才能稳定保证封面、分页、Reference 新页和悬挂缩进
- 引用核验报告最终由后端统一生成结构化 PDF，不再交付 `.txt`
- 课号不会新增前端输入框，而是在首次生成大纲时从任务标题、特殊要求、材料里自动提取并保存
- 前端 Supabase 地址和公开 key 只允许从环境变量读取，不再在源码里保留真实线上兜底值
- 工作台轮询不再一直固定频率死问，而是按阶段逐步放慢，并带总超时
