# 拼代代 — 设计蓝图

## 产品是干什么的

拼代代是一个帮用户写英文文章的网页工具。用户上传自己的材料，系统帮忙生成英文大纲和正文，最后还能做"降低 AI 痕迹"的处理。

## 当前线上架构：三层分工

整个系统分成三层，每层只管自己的事：

```
┌─────────────────────────────────────────┐
│  第一层：用户看到的网页（前端）              │
│  技术：React + TypeScript + Tailwind       │
│  部署：Railway 前端服务                       │
│  职责：展示界面、收集用户输入、显示结果       │
│  红线：不做业务判断，不存业务真相             │
└────────────────┬────────────────────────┘
                 │ 调用接口
┌────────────────▼────────────────────────┐
│  第二层：业务处理服务（后端）                │
│  技术：Express (Node.js)                    │
│  部署：Railway app + cleanup                 │
│  职责：处理业务逻辑、调用 AI、管理文件       │
│  红线：不自建登录系统，不让前端自己算账       │
└────────────────┬────────────────────────┘
                 │ 读写数据
┌────────────────▼────────────────────────┐
│  第三层：数据和认证（基础设施）              │
│  技术：Supabase（数据库 + 认证 + 文件存储）  │
│  职责：存数据、管用户账号、存文件、做行级权限门禁 │
│  红线：数据真相只有这一份，不允许其他地方存副本│
└─────────────────────────────────────────┘
```

## 用户操作的完整流程图

```
用户注册
  ↓
Supabase 创建账号
  ↓
Railway 初始化（创建用户档案 + 钱包）← 必须成功才能进工作台
  ↓
用户登录
  ↓
兑换激活码获得积分
  ↓
创建任务 + 上传材料
  ↓
系统生成首版大纲（不收费）
  ↓
用户可修改大纲（最多改 4 次，不收费）
  ↓
用户确认大纲 → 冻结积分 → 开始收费链路
  ↓
正文生成 → 字数矫正 → 引用核验 → 交付整理
  ↓
成功：扣除积分，用户下载文件
失败：退回冻结积分，记录失败原因
  ↓
（可选）用户发起降 AI → 再次冻结积分 → 处理 → 成功扣费/失败退回
```

## 核心设计决策（为什么这么做）

### 决策 1：前端不存任何真实业务状态

**为什么**：如果前端自己记"任务到第几步了"，一旦用户刷新页面或换个浏览器，数据就丢了。所以所有状态都存在后端数据库里，前端每次都去问后端"现在到哪了"。

### 决策 2：登录只用 Supabase Auth，不自己做

**为什么**：自己做登录系统太容易出安全问题（密码泄露、被攻击等）。Supabase 是专业做这个的，比我们自己做靠谱得多。Railway 的业务接口也用 Supabase 的登录凭证来验证身份，不搞两套。

### 决策 3：同一时间一个用户只能做一个任务

**为什么**：多任务并行会让状态管理变得极其复杂。第一阶段先保证一个任务跑通跑稳，以后再扩展。而且这个限制在前端、后端、数据库三层都要拦截，防止任何一层漏掉。

### 决策 4：工作台是一个页面多个状态，不是多个页面

**为什么**：用户从头到尾在同一个页面（`/dashboard/workspace`）操作，页面根据任务进度自动显示不同内容。这样用户不会迷路，开发也不会做一堆页面跳来跳去搞混。

### 决策 5：收费阶段失败必须自动退款

**为什么**：用户花了钱但没拿到结果，必须立刻把钱退回去。不能让运营手动退，因为运营可能忙、可能忘。系统必须自动处理，而且退款必须有流水记录，账目要对得上。

### 决策 6：第一阶段不做完整运营前台页面，但要预留运营接口

**为什么**：完整后台页面很费时间。第一阶段先把运营需要的后端接口和数据库操作留好，必要时可以直接用 Supabase 和 Railway 配合处理，后面再决定要不要补页面。

### 决策 7：涉及钱和任务状态的关键动作，尽量收进数据库里一次做完

**为什么**：像“兑换激活码”“确认大纲开始收费”“发起降 AI”这种动作，一旦被连续点两次，或者两个请求同时撞上来，就很容易出现重复扣费、状态错乱、半成功半失败。现在这类关键动作统一收进数据库函数里，一次完成检查条件、改余额、写流水、改任务状态，避免中途卡住。

## 数据库核心表（简化说明）

| 表名 | 存什么 |
|------|--------|
| `user_profiles` | 用户基本信息（昵称、状态等） |
| `wallets` | 用户钱包（余额） |
| `credit_ledger` | 积分流水（充值、消费、退款的每一笔记录） |
| `recharge_codes` | 激活码（面值、状态、谁用了） |
| `tasks` | 任务主记录（当前阶段、状态、失败原因等） |
| `task_files` | 任务相关的文件（上传的材料、生成的结果） |

补充规则：

- 面向用户的业务表现在都开了 RLS（行级门禁），用户只能看到自己的数据
- `recharge_codes`、`system_config` 这类敏感表不允许公开身份直接读取，只能走后端 service role

## 文件存储规则

所有文件存在 Supabase Storage 里，按任务 ID 分文件夹。用户上传的材料和系统生成的结果文件都在这里，通过签名链接给用户下载，链接有过期时间。

## 当前部署落点

- Railway 项目：`glistening-achievement`
- 前端服务：`拼代代前端`
- 后端服务：`app`
- 清理服务：`cleanup`
- Supabase：负责 Auth、Database、Storage
- GitHub：仓库已接好自动发布，推送 `main` 会自动发到 Railway

## 环境变量

敏感信息（数据库地址、AI 密钥、Supabase 密钥等）全部通过环境变量注入，不允许提交到公开仓库。参考各目录下的 `.env.example` 文件。

补充说明：

- 主写作链路当前统一通过 `OPENAI_MODEL=gpt-5.4` 读取模型名
- 降 AI 当前单独走 Undetectable Humanization API，不再复用 OpenAI
- 降 AI 固定参数：`model=v11sr`、`strength=More Human`、`readability=University`、`purpose=Essay`
