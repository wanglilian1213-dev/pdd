# Lessons

- 2026-05-11：改线上模型变量前，必须先确认线上正在跑的代码已经支持该模型；如果只改变量不先部署兼容代码，会导致后台启动失败。
- 2026-05-11：凡是会影响线上启动的配置变更，改完必须立刻看部署状态、启动日志和真实网页，而不是只做本地测试。
- 2026-05-11：第三方检测服务即使登录和上传都正常，也可能在模型版本接口内部报错；真实验收必须覆盖结果页和后台完成状态，不能只看“提交成功”。

- 2026-05-19：部署相关方案不能只照旧文档下结论。用户指出当前已不用 Vultr、全在 Railway；以后涉及线上架构必须先核对真实当前部署，再给方案。
- 2026-05-19：第三方 Google 登录流程不要只按旧英文文案或单一中文翻译写选择器；实际页面可能显示“试试其他方式”。以后涉及登录分支必须用真实页面截图 / 文本复核后再固化识别。
- 2026-05-25：设计医学/工程图示流程时，材料缺精确参数不能直接降级；必须先判断是否适合联网补充。适合联网且作业允许时，先找可靠来源并写入 References；不适合联网或来源不可靠时，才降级为可验证高层图示或要求补材料。
- 2026-05-25：当用户要求“100% 把握”时，不能口头保证绝对正确；要把策略拆成可被打穿的真实漏洞，并把每个漏洞变成 PRD 规则、测试红线、发布门槛或线上报警。尤其要覆盖隐私泄露、材料误导指令、环境不一致、状态原子性和测试集过时。
- 2026-05-25：用户要求测试 pindaidai 网站时，不能把底层模型服务 pddapi 测通就当完成；必须回到 pindaidai 的真实页面和真实业务流程，覆盖登录、写作、修改、评审、AI 检测/降 AI、历史、充值和下载。
- 2026-05-26：用户要求“网站能不能应付所有场景”时，不能把本地题库、单元测试或模型接口测试说成网站全流程通过；必须分清真实生产网页已跑、上传/付费长任务未跑、本地自动防线已补三类证据。
- 2026-05-27：当用户再次强调“专注 pindaidai 网站”时，最终汇报必须按网站功能和交付质量来写：哪些线上页面真实打开了，哪些无文件/余额为 0 的保护被验证了，哪些质量硬闸门在本地通过了，哪些上传到下载的生产长流程还没有证据；不要把模型服务或单元测试包装成网站全通过。
- 2026-05-27：如果 Chrome/浏览器工具卡住，不能用“工具问题”含糊带过；要具体说明哪些部分已确认正常（Chrome、插件、本机桥接文件）和真正卡住的位置（控制通道超时、无法上传/下载验收），并继续把生产长流程列为未完成证据缺口。
- 2026-05-27：真实生产写作长流程失败时，不能只看页面上的笼统失败文案；必须查询任务事件里的技术原因，再把原因变成可复现测试。模型输出字段不能假设一定存在，缺字段应按安全默认值处理，同时保留真实冲突必须停下的规则。
- 2026-05-31：用户要求修复计划“细到哪一行、哪个链路”时，计划不能只写模块级方向；必须先定位到具体页面、接口、服务函数、数据流断点和要验证的证据，再进入执行。
- 2026-06-01：用户要求梳理“整个流程逻辑”时，不能只回答直接断点；必须沿完整任务流水线看每个会改写内容的环节，指出哪些地方有同类风险，再把修复做成统一边界，避免局部补丁制造新冲突。
