## 1. 文档说明

### 1.1 这份文档是干什么的

这份 PRD 用来指导「拼代代」第一阶段正式落地。

它不是一份“脑暴记录”，也不是一份“理想化愿景文档”。  
它是一份**能直接拿去开工、分工、验收**的产品需求文档。

### 1.2 这次 PRD 重写的前提

这次不是从零设计一个新产品。  
这次的现实前提已经非常清楚：

1. 用户端前端已经做出来了。
2. 用户端的页面结构、路由结构、主流程顺序，已经基本定型。
3. 这套前端必须保留，不能推翻重做。
4. 后续开发团队和运营团队都没有编程基础，主要依赖 AI 辅助维护。

所以，这份文档的核心任务不是“重新定义用户端应该长什么样”，而是：

`在保留现有前端的前提下，把账号、收费、任务、文件、运营、部署这些真实业务能力接上去。`

### 1.3 第一阶段最重要的原则

第一阶段必须同时满足下面 5 条：

1. 用户端继续使用现有前端。
2. 业务规则只能有一套解释，不能前后打架。
3. 状态真相只能有一个来源。
4. 运营必须看得懂任务，不靠猜。
5. 整体结构必须适合 0 代码团队长期维护。

## 2. 产品定义

### 2.1 产品一句话定义

拼代代是一个网页写作工具。  
用户上传材料后，系统自动生成英文大纲、英文正文、引用核验结果，并支持后续继续做降 AI 处理。

### 2.2 产品要解决的问题

用户在做英文论文、英文报告、英文分析类写作时，通常会遇到下面这些问题：

1. 材料很多，整理成本高。
2. 不知道怎么先搭出一个像样的大纲。
3. 正文很耗时间，字数也难控制。
4. 引用格式容易出错。
5. 写完以后还可能需要继续做“降 AI”处理。

拼代代第一阶段要解决的，不是所有写作问题，而是下面这个最核心的闭环：

`从上传材料，到拿到可下载的最终文件，再到按需要继续做降 AI。`

### 2.3 第一阶段的产品定位

第一阶段不是一个“大而全的平台”，而是一个**能稳定跑通商业闭环的交付型产品**。

第一阶段最看重的不是“功能很多”，而是下面这些事情必须成立：

1. 用户能自己注册登录。
2. 用户能自己充值额度。
3. 用户能发起任务。
4. 用户能看到大纲并确认。
5. 用户能拿到最终文件。
6. 用户能继续做降 AI。
7. 失败时系统能解释清楚，并自动做该做的退款。

## 3. 第一阶段目标与非目标

### 3.1 第一阶段目标

第一阶段目标固定为：

1. 保留现有用户端页面结构和主流程。
2. 用最少的平台把真实业务能力接起来。
3. 让“生成正文”和“降 AI”都能按统一规则收费。
4. 让任务状态、文件状态、收费状态都变成真实数据。
5. 让运营能看懂用户、充值、任务和失败原因。
6. 让系统可以正式上线，而不是只做演示。

### 3.2 第一阶段不做

第一阶段明确不做下面这些内容：

1. 不重做用户端页面主结构。
2. 不做站内在线支付。
3. 不做邮箱验证。
4. 不做用户自助找回密码。
5. 不做复杂角色权限系统。
6. 不做完整重型自研后台。
7. 不做同一用户多任务并行。
8. 不做复杂会员价、活动价、阶梯价。
9. 不做用户上传自定义正文后再降 AI。
10. 不做独立手机端产品设计。
11. 不做多个自动化平台拼接主流程。

### 3.3 第一阶段成功标准

第一阶段成功，不是看页面漂不漂亮，而是看下面这些是否成立：

1. 用户能独立完成从注册到交付的完整闭环。
2. 用户充值、消费、退款能对得上。
3. 用户刷新页面后不会丢任务进度。
4. 失败时运营不用猜，就知道发生了什么。
5. 开发改功能时，不会因为结构太散而越改越乱。

### 3.4 第一阶段核心指标

第一阶段上线后，至少要持续看下面这些指标：

1. 注册成功率  
   看有多少用户能顺利完成注册和初始化。

2. 从注册到首次创建任务的转化率  
   看用户注册后有没有真的开始使用产品。

3. 大纲生成成功率  
   看第一步核心能力是否稳定。

4. 正文链路成功率  
   看“确认大纲到最终交付”这条最重要收费链路是否稳定。

5. 自动退款成功率  
   看失败任务发生时，系统有没有把该退的钱真的退回去。

6. 运营可判读率  
   看运营遇到失败任务时，是否能直接判断“现在在哪一步、为什么失败、下一步找谁”。

## 4. 用户角色

### 4.1 普通用户

普通用户通过网页完成下面这些动作：

1. 注册
2. 登录
3. 查看余额
4. 兑换激活码
5. 创建任务
6. 上传材料
7. 查看和修改大纲
8. 确认大纲并启动正文生成
9. 查看任务进度
10. 下载结果文件
11. 发起降 AI
12. 查看历史任务

### 4.2 运营

第一阶段的运营不是“后台系统管理员”，而是“看数据、做少量必要操作的人”。

运营主要做这些事：

1. 看用户状态和余额
2. 看充值记录和激活码状态
3. 看任务当前进度和失败原因
4. 看是否已退款
5. 批量生成激活码
6. 作废未使用激活码
7. 禁用或恢复账号
8. 调整少量业务配置

### 4.3 运营权限规则

第一阶段不做复杂权限层级。  
直接采用最简单、最稳的规则：

1. 普通用户只能访问用户端网页。
2. 运营查看数据主要通过 Supabase 后台。
3. 运营涉及“写入类动作”时，必须通过 Railway 的内部运营入口。
4. 这个内部运营入口只对运营白名单邮箱开放。
5. 普通用户即使已登录，也不能执行任何运营写动作。
6. 第一阶段运营白名单固定为 1 到 2 个邮箱，不做后台可视化管理。
7. 如需增删运营白名单邮箱，由开发在 Railway `app` 的环境变量中修改并重新部署。
8. 第一阶段密码重置是唯一允许运营直接在 Supabase Auth 后台执行的写动作。

## 5. 核心产品闭环

### 5.1 核心闭环一句话

第一阶段核心闭环固定为：

`注册登录 -> 充值 -> 创建任务 -> 生成大纲 -> 确认大纲 -> 生成正文 -> 下载结果 -> 继续降 AI`

### 5.2 闭环里最关键的两个收费动作

第一阶段只有两个收费动作：

1. 生成正文
2. 降低 AI

除此之外，其他动作都不收费。

### 5.3 闭环里最关键的两个失败处理原则

1. 不收费阶段失败：只记录失败，不涉及退款。
2. 收费阶段失败：必须自动退回本次冻结积分。

## 7. 工作台详细需求

### 7.1 工作台总原则

工作台必须满足下面这些要求：

1. 用户全程只在 `/dashboard/workspace` 完成主流程。
2. 同一时间一个用户只能有一个进行中的任务。
3. 页面刷新后必须恢复到真实阶段。
4. 页面上的进度步骤必须来自真实后端状态，不允许前端自己猜。
5. 页面顶部的用户信息和余额必须是真实数据。

### 7.2 工作台 7 步结构

保留现有 7 步视觉结构：

1. 上传分析
2. 确认大纲
3. 正文生成
4. 字数矫正
5. 引用核验
6. 交付核验
7. 降低 AI

### 7.3 步骤 1：上传分析

页面必须包含：

1. 文件上传区域
2. 已上传文件列表
3. 特殊要求输入框
4. 开始分析并生成大纲按钮

上传规则：

1. 第一阶段默认开放下面这些上传格式：
   - `txt`
   - `md`
   - `docx`
   - `pdf`
   - `ppt`
   - `pptx`
   - `jpg`
   - `jpeg`
   - `png`
   - `webp`
   - `gif`
   - `heic`
   - `heif`
   - `bmp`
   - `tiff`
   - `tif`
2. 单个任务最多 10 个文件。
3. 单个文件默认不超过 20MB。
4. 单个任务总大小默认不超过 50MB。
5. 第一阶段必须默认开放上面这些格式，不再做“先关掉某些格式再说”的临时规则。
6. 演示文稿和图片类材料进入系统后，后端必须先做统一材料解析，再进入大纲阶段。
7. 统一材料解析的规则是：
   - 能直接抽出文字的文件，优先抽出文字
   - 图片类文件走 OCR 或视觉识别，把可用内容转成统一材料文本
   - `ppt/pptx` 先抽出页面文字，再补充页面结构信息
8. 如果某个文件损坏、无法识别或解析失败，系统必须在大纲阶段给出明确失败原因，而且该阶段不收费。

这里必须补一条统一规则：

`页面展示的可上传格式、后端真实开放格式、报错提示里的格式说明，必须来自同一套当前生效规则。`

### 7.4 步骤 2：确认大纲

页面必须包含：

1. 英文大纲正文
2. 目标字数
3. 引用格式
4. 修改意见输入框
5. 重新生成大纲
6. 确认大纲并开始生成

这里的规则是：

1. 首版大纲生成后，用户进入确认大纲阶段。
2. 用户可以继续修改大纲。
3. 用户可以确认目标字数和引用格式。
4. 用户确认后，系统才允许进入收费正文链路。

### 7.5 步骤 3 到 5：正文处理中

这几步继续在同一工作台里展示，不拆成新页面。

真实阶段包括：

1. 正文生成
2. 字数矫正
3. 引用核验

页面必须做到：

1. 展示当前真实阶段
2. 自动刷新状态
3. 支持手动刷新
4. 失败时直接显示失败卡片

### 7.6 步骤 6：交付核验

页面必须展示：

1. 最终版文章下载
2. 引用核验报告下载
3. 自动降 AI 入口
4. 人工降 AI 联系入口

### 7.7 步骤 7：降低 AI

页面必须展示：

1. 降 AI 处理中
2. 降 AI 成功后的最新下载入口
3. 再次发起降 AI 的入口

## 8. 我的任务页详细需求

### 8.1 页面定位

这页不是让用户处理复杂异常的。  
它的主要作用是让用户快速看懂：

1. 自己有哪些任务
2. 当前任务是进行中、已交付还是失败
3. 哪些结果还能下载

### 8.2 状态规则

用户侧任务状态统一压缩为：

1. 处理中
2. 已交付
3. 生成失败

这里必须把降 AI 处理中写死：

`只要任务当前处于 humanizing，也就是降 AI 处理中，任务列表统一显示为“处理中”；只有 humanizing 结束后，才回到“已交付”。`

### 8.3 失败任务按钮规则

第一阶段失败任务右侧按钮的含义固定为：

`查看失败原因`

行为固定为：

1. 不在任务列表里直接展开复杂失败详情
2. 点击后统一跳到：
   - `/dashboard/workspace?taskId={任务编号}`
3. 工作台按这个 `taskId` 打开该任务的只读失败卡片
4. 只读失败卡片必须展示：
   - 失败阶段
   - 失败原因
   - 是否已退款
   - 建议下一步动作

第一阶段不提供：

1. 一键重跑
2. 一键重新收费
3. 在原任务上直接继续跑

### 8.4 已交付任务按钮规则

第一阶段已交付任务右侧按钮的含义固定为：

`查看结果`

行为固定为：

1. 不在任务列表里直接下载文件
2. 点击后回到工作台，并在当前工作台地址后带上任务编号
3. 第一阶段固定采用：
   - `/dashboard/workspace?taskId={任务编号}`
4. 用户在结果区里再选择具体要下载的文件
5. 打开历史任务结果时，工作台进入“指定任务只读结果模式”
6. 这个只读结果模式只能看结果、失败原因和下载入口，不能在历史任务上继续修改大纲或重新发起正文链路

这样做的原因是：

1. 一个已完成任务可能同时存在多个可下载文件
2. 任务列表页不适合承担复杂文件选择逻辑
3. 对不会编程的团队来说，统一回结果页最简单，也最不容易做错
4. 不新增新页面，也不破坏当前前端主结构

## 10. 注册、登录与账号规则

### 10.1 注册规则

第一阶段是真注册，不是假入口。

用户注册流程固定为：

1. 用户提交邮箱和密码
2. Supabase 创建账号
3. 前端拿到登录态
4. 前端调用 Railway 的用户初始化接口
5. Railway 幂等创建：
   - `user_profiles`
   - `wallets`
6. 初始化成功后，才进入工作台

### 10.2 初始化失败规则

如果账号创建成功了，但用户初始化失败：

1. 不允许进入业务流程
2. 页面提示“账号初始化失败，请稍后重试”
3. 不允许出现“注册成功但钱包不存在”的半成品账号

### 10.3 登录规则

1. 登录只认 Supabase Auth。
2. 业务接口身份识别也只认同一套 Supabase 登录态。
3. Railway 不自建第二套登录系统。

### 10.4 忘记密码规则

第一阶段不做用户自助找回密码。

固定处理方式如下：

1. 用户在登录页看到“忘记密码请联系客服”的提示
2. 用户联系人工客服或运营
3. 运营在 Supabase Auth 后台手动协助用户重置密码
4. 重置完成后，用户使用新密码重新登录

这里必须写死：

1. 不做站内“发邮件找回”
2. 不做站内“短信找回”
3. 不让运营直接改数据库密码字段
4. 这是第一阶段唯一允许绕开 Railway 内部运营入口、直接在 Supabase Auth 后台执行的写动作

### 10.5 禁用账号规则

第一阶段“账号禁用”的定义固定为：

1. 即使认证成功，也不能进入任何业务页面。
2. 不能创建任务。
3. 不能兑换激活码。
4. 不能下载文件。
5. 不能发起降 AI。

### 10.6 第一阶段安全兜底

第一阶段至少加这 3 类限频：

1. 注册接口限频
2. 登录失败尝试限频
3. 激活码连续试错限频

---

## 11. 任务规则

### 11.1 单任务规则

第一阶段固定规则：

`同一用户同一时间只能有一个进行中的任务。`

这个规则必须三层一起兜住：

1. 前端提示并拦截
2. Railway 创建任务时再次检查
3. 数据库层加硬限制，不允许插入第二条进行中任务

### 11.2 创建任务规则

创建任务顺序固定为：

1. 检查用户状态
2. 检查是否已有进行中任务
3. 检查文件数量和大小
4. 保存任务主记录
5. 保存材料文件记录
6. 把任务状态写成 `outline_generating`
7. 启动首版大纲生成

### 11.3 任务状态真相

任务状态只有一个真相来源：Supabase 主数据库。

前端不保存第二份真状态。  
运营视图也不保存第二份真状态。  
所有人都读同一份任务结果。

### 11.4 任务失败规则

每次失败都必须留下：

1. 失败阶段
2. 失败原因
3. 是否已退款
4. 建议下一步动作

---

## 12. 大纲规则

### 12.1 首版大纲

首版大纲生成后，系统同时产出：

1. 英文大纲
2. AI 识别的目标字数
3. AI 识别的引用格式

### 12.2 默认值规则

如果 AI 没识别出来：

1. 目标字数默认 `1000`
2. 引用格式默认 `APA 7`

### 12.3 修改大纲规则

大纲修改定义为：

`基于当前最新版大纲和本次修改意见，生成下一版大纲。`

不是从零重开任务。

### 12.4 修改次数规则

1. 首版大纲不算修改次数
2. 后续最多修改 4 次
3. 修改次数用尽后，不能继续改
4. 修改目标字数和引用格式，不占用这 4 次

### 12.5 确认大纲规则

确认大纲并开始正文时，必须一次性完成：

1. 锁定大纲版本
2. 锁定目标字数
3. 锁定引用格式
4. 计算应收积分
5. 冻结积分
6. 状态改为 `writing`

不能成功一半。

---

## 13. 正文主流程规则

### 13.1 正文主流程定义

正文主流程是第一阶段最核心的收费链路，包含：

1. 正文生成
2. 字数矫正
3. 引用核验
4. 交付整理

这四步算一次收费动作，不拆开分别收费。

### 13.2 收费口径

正文生成按“用户最终确认的目标字数”计费。  
不是按最后生成出来到底多少字收费。

### 13.3 正文流程成功条件

只有同时满足下面这些，才算正文链路成功：

1. 正文生成完成
2. 字数校正完成
3. 引用核验完成
4. 最终文件生成成功
5. 文件记录写入成功
6. 收费结算成功

### 13.4 正文流程失败规则

正文链路任意阶段失败，都必须：

1. 记录失败
2. 写清原因
3. 退回冻结积分
4. 写退款流水

### 13.5 交付结果

正文链路成功后，用户必须能拿到：

1. 最终正文 `.docx`
2. 引用核验报告

---

## 14. 降 AI 规则

### 14.1 降 AI 定位

降 AI 不是正文主流程的必经步骤。  
它是正文完成后的独立附加收费链路。

### 14.2 唯一入口

降 AI 唯一入口固定在工作台结果区。  
第一阶段不在其他页面另开新入口。

### 14.3 前置条件

发起降 AI 前必须满足：

1. 任务已完成
2. 有可用正文版本
3. 当前没有降 AI 在处理中
4. 当前余额足够

### 14.4 输入版本规则

1. 第一次降 AI -> 使用原始最终正文
2. 第二次降 AI -> 使用上一次成功降 AI 的版本
3. 后续依次类推

### 14.5 收费口径

降 AI 按“本次输入正文的实际字数”收费。

### 14.6 状态规则

1. 降 AI 处理中 -> 任务主状态为 `humanizing`
2. 降 AI 成功 -> 回到 `completed`
3. 降 AI 失败 -> 也回到 `completed`

这里必须写清楚原因：

`因为降 AI 是附加动作，不应该把已经完成的正文主任务重新打成失败。`

### 14.7 失败规则

降 AI 失败时：

1. 只退回本次费用
2. 不破坏已有正文结果
3. 不影响主任务已完成事实
4. 运营视图必须仍然高亮这次失败

---

## 15. 收费与钱包规则

### 15.1 第一阶段收费动作只有两个

1. 生成正文
2. 降低 AI

### 15.2 第一阶段不收费动作

至少包括：

1. 注册
2. 登录
3. 创建任务
4. 首版大纲生成
5. 修改大纲
6. 查看任务
7. 下载有效期内文件

### 15.3 统一计费单位

所有收费统一按：

`每 1000 英文单词为一个计费单位，向上取整`

这里必须明确：

1. 计费单位是“英文单词”，不是“字符数”
2. 第一阶段所有价格配置、预估价格、扣费结果都按英文单词计算

### 15.4 价格配置

第一阶段只保留两个价格配置项：

1. 正文单价
2. 降 AI 单价

### 15.5 统一账务模型

所有收费动作统一走：

1. 检查余额
2. 冻结积分
3. 执行任务
4. 成功结算
5. 失败退回

### 15.6 可见流水类型

对用户和运营展示的流水只保留 3 类：

1. 充值
2. 消费
3. 退回

## 16. 激活码规则

### 16.1 第一阶段唯一充值方式

第一阶段不做在线支付。  
用户线下或人工购买激活码，再在网页内兑换。

### 16.2 兑换规则

一次成功兑换，必须同时完成：

1. 校验激活码
2. 增加余额
3. 写充值流水
4. 把激活码标记为已使用

不能成功一半。

### 16.3 激活码状态

激活码只有三种状态：

1. 未使用
2. 已使用
3. 已作废

### 16.4 运营相关规则

第一阶段：

1. 批量生成激活码要走 Railway 内部运营入口
2. 作废未使用激活码也要走 Railway 内部运营入口
3. 不能靠运营直接改数据库表

## 17. 文件规则

### 17.1 文件类型

第一阶段涉及两类文件：

1. 用户上传材料
2. 系统生成结果

### 17.2 结果文件类型

至少包括：

1. 最终正文文件
2. 引用核验报告
3. 降 AI 结果文件

### 17.3 下载规则

下载前必须先校验：

1. 文件是否属于当前用户
2. 文件是否还没过期
3. 文件记录是否有效

### 17.4 文件过期规则

1. 结果文件有过期时间
2. 过期后按钮置灰并提示无法下载
3. 文件可删，但记录要保留

### 17.5 文件清理规则

1. 结果文件到期后自动清理
2. 材料文件按规则清理
3. 清理动作必须同步更新记录

## 18. 运营侧需求

### 18.1 第一阶段运营总体原则

第一阶段不做重型自研后台。  
运营结构固定为：

1. Supabase 后台：负责看
2. Railway 内部运营入口：负责做少量必要写动作

### 18.2 运营核心入口

第一阶段固定 4 个核心入口：

1. 用户视图
2. 充值视图
3. 任务监控视图
4. 配置视图

### 18.3 用户视图

至少展示：

1. 邮箱
2. 注册时间
3. 账号状态
4. 当前余额
5. 是否有进行中任务

### 18.4 充值视图

至少展示：

1. 激活码
2. 面值
3. 当前状态
4. 使用人
5. 使用时间

### 18.5 任务监控视图

这是第一阶段最重要的运营视图。

必须一眼看到：

1. 任务编号

4. 当前状态

即使任务主状态已经回到 `completed`，如果最近一次降 AI 失败，任务监控视图仍然必须单独高亮显示：

1. 最近一次降 AI 失败
2. 该次是否已自动退款
3. 当前建议动作

### 18.7 配置视图

第一阶段只保留少量可调配置：

6. 材料保留天数
7. 卡住任务超时时间

### 18.8 第一阶段默认配置

第一阶段默认值先写死为：

1. 正文单价：每 1000 词 `250` 积分
2. 降 AI 单价：每 1000 词 `250` 积分

4. 材料保留天数：`3` 天
5. 卡住任务超时时间：`30` 分钟

这几个值虽然属于可配置项，但第一阶段必须先带着这套默认值落地、测试和验收。

### 18.9 运营不能做的事

运营不能直接：

1. 改余额
2. 改任务主状态
3. 补扣积分
4. 把失败任务改成成功

## 19. 系统方案

### 19.1 平台方案

第一阶段固定使用两个平台：

1. Railway
2. Supabase

### 19.2 Railway 职责

Railway 内固定有三个运行单元：

1. `web`
   负责部署现有前端
2. `app`
   负责所有业务规则和 AI 调用
3. `cleanup`
   负责文件清理和卡住任务兜底

### 19.3 Supabase 职责

Supabase 固定负责：

1. Auth
2. Database
3. Storage
4. Dashboard

### 19.4 连接规则

前端只允许两种连接：

1. 登录相关动作 -> Supabase Auth
2. 其他业务动作 -> Railway app

不允许第三种连接方式。

### 19.5 唯一业务裁判

Railway `app` 是唯一业务裁判。  
收费、退款、任务推进、失败判断，都只能在这里做。

### 19.6 唯一真相中心

Supabase 主数据库是唯一业务真相中心。  
任务状态、钱包、文件记录、失败原因，都只认这里。

## 26. 最终结论

这份 PRD 的核心，不是追求“技术上最酷”或者“形式上最统一”。  
它真正要解决的是下面这件事：

`在必须保留现有前端的前提下，用尽量少的系统、尽量清楚的规则，做出一个真正能注册、能收费、能交付、能维护的正式产品。`

第一阶段只要守住下面这几条，就不会走偏：

1. 用户端不重做
2. 业务规则只认 Railway app
3. 数据真相只认 Supabase 数据库
4. 收费只走一套统一规则
5. 失败必须让运营看得懂
6. 所有设计都优先服务于低认知负担

这就是第一阶段拼代代应该长成的样子。
